SUMMARY

Contexto

Context

Las apps de transferencias viven de operaciones repetidas: buscar un contacto, verificar un CBU, confirmar un monto. Cada inconsistencia visual en ese camino es fricción y desconfianza. Vault Transfer DS nace para sistematizar ese lenguaje: un design system atómico donde cada componente está pensado para flujos transaccionales con dinero real.

Transfer apps live on repeated operations: search a contact, verify a bank account, confirm an amount. Every visual inconsistency along that path is friction and distrust. Vault Transfer DS was born to systematize that language: an atomic design system where every component is built for transactional flows with real money.

Objetivo

Objective

Construir una librería escalable — tokens, 25 componentes y 450+ variantes — que cubra el happy path completo de una transferencia, con estados explícitos (default, pressed, focused, disabled, loading, success) y accesibilidad como requisito de base, no como retrofit.

Build a scalable library — tokens, 25 components, 450+ variants — that covers the complete happy path of a transfer, with explicit states (default, pressed, focused, disabled, loading, success) and accessibility as a baseline requirement, not a retrofit.

APRENDIZAJES

LEARNINGS

Lo que aprendí

What I learned

Los estados son el sistema

States are the system

Definir 7 estados por componente desde el día uno evitó los "casos especiales" que suelen romper un DS cuando llega el desarrollo.

Defining 7 states per component from day one avoided the "special cases" that usually break a DS once development starts.

Props antes que variantes

Props before variants

Modelar Contact List Item como State × AvatarType × Bank × ShowSecondary (72 combinaciones) mantiene una sola fuente de verdad en lugar de 72 dibujos sueltos.

Modeling Contact List Item as State × AvatarType × Bank × ShowSecondary (72 combinations) keeps a single source of truth instead of 72 loose drawings.

El flujo valida la librería

The flow validates the library

Armar el happy path completo con piezas del sistema fue el mejor test de cobertura: cada hueco que apareció se convirtió en backlog del evolutivo.

Building the complete happy path with system pieces was the best coverage test: every gap that surfaced became backlog for the next iteration.
